# מבנה הבוט — מה הוא עושה ואיך כל לשונית עובדת

מסמך עזר מסכם לפיתוח עתידי. מתעדכן ידנית כשמשנים תכונה משמעותית.

---

## 1. סקירה כללית — מה הבוט עושה

הבוט סוחר ב-**Polymarket** על שווקי **BTC Up/Down** קצרי-טווח: שאלות בינאריות בנוסח
"האם מחיר BTC בעוד 5/15 דקות יהיה גבוה מהמחיר עכשיו?". כל שוק מציע שני חוזים —
**Up** ו-**Down** — שמחירם נע בין `0.01$` ל-`0.99$` (סנט אחד עד 99 סנט). אם הצד שלך
מנצח, החוזה משלם `1.00$`; אחרת — `0$`.

המערכת מתחלקת לשני רכיבים:

- **Engine** (`engine/`) — FastAPI + uvicorn ב-Python; חיבור WebSocket ל-Polymarket,
  שמירת state מקומי, חישוב סיגנלים טכניים, ניהול פוזיציות, ביצוע הזמנות.
- **Frontend** (`src/`) — React + Vite; ניווט בין 9 לשוניות, גרפים בזמן אמת, טפסי
  הגדרות, וכפתורי שליטה.

**מצבי הפעלה:**

| מצב | מה זה אומר |
|------|-------------|
| **דמו (Demo)** | מבצע "כאילו" פקודות מול ספר ההזמנות האמיתי, אבל **לא** משדר לבלוקצ׳יין. יתרה וירטואלית. |
| **מסחר חי (Live)** | חתימה אמיתית עם המפתח שלך, ביצוע על הבלוקצ׳יין. דורש USDC ב-CLOB של Polymarket. |
| **חצי-אוטומטי** | המנוע מציע עסקאות; המשתמש מאשר ידנית בכפתור. |
| **אוטומטי מלא** | המנוע מבצע ללא אישור, לפי האסטרטגיה המוגדרת. |

המבנה מבטיח שאפשר לרוץ בדמו כמה שרוצים, ורק כשבוטחים — להפעיל לייב באותו UI.

### 3.1 לוח בקרה (`dash`)

**מטרה:** המסך הראשי בזמן ריצה. רואים את השוק הנוכחי, מחירי BTC, מחירי החוזים,
פוזיציות פתוחות, וכפתורי שליטה.

**אזורים עיקריים:**

1. **כרטיס שוק נוכחי** ([App.tsx:2510-2620](../src/App.tsx#L2510)):
   - שם השוק (slug + epoch)
   - זמן שנותר לסיום החלון (count-down)
   - מחיר ייחוס BTC לפתיחת החלון (Chainlink)
   - מחיר BTC עכשיו + הפרש מהפתיחה
   - אינדיקטור **WS Live** (ירוק/אדום) — מאשר שהזרימה פעילה
   - mid/bid/ask לחוזי Up + Down

2. **גרף BTC חי** — recharts LineChart, מחירי BTC במהלך החלון הנוכחי. מתעדכן
   כל ~1s מ-`/api/btc/live` (Chainlink polygon RPC כמקור-אמת + Binance fallback).

3. **גרף מחיר חוזה (Up / Down)** — bid/ask לאורך זמן. מקור: WebSocket
   `wss://...polymarket.com/ws/market` דרך ה-proxy ב-[engine/ws_price_stream.py](../engine/ws_price_stream.py).
   מאז התיקון האחרון, יש watchdog שמכריח reconnect אחרי 30s ללא הודעה
   ו-dedup-by-token שלא מאבד עדכוני מחיר.

4. **כפתור הפעלה/השהיה של ה-StrategyRunner** — שולח POST ל-`/api/strategy/mode`
   עם `off`/`manual`/`auto`.

5. **פוזיציות פתוחות** — לכל פוזיציה: צד (Up/Down), מספר חוזים, מחיר ממוצע,
   PnL לא ממומש, peak/trough של הפוזיציה.

6. **עסקאות אחרונות** — טבלה מ-`demo_state.trades` (15 אחרונות).

**מקורות נתונים:**
- `GET /api/market/current` — שוק פעיל
- `GET /api/market/orderbook-summary` — bid/ask משולב (WS אם טרי, אחרת fallback)
- `GET /api/btc/live` — מחיר BTC
- `GET /api/demo/state` — יתרה, פוזיציות, עסקאות אחרונות
- `WS /ws/prices` — push זמן-אמת ל-bid/ask

### 3.2 אסטרטגיה (`strategy`)

**מטרה:** הגדרת התנהגות ה-StrategyRunner — מתי להיכנס, כמה, איך לצאת.

**המסך מכיל ~20 שדות הגדרה** ([App.tsx:2951-2950+](../src/App.tsx#L2951)):

#### בלוק "תקציב"

| שדה | משמעות |
|------|---------|
| **מצב סכום השקעה** | `fixed` (סכום קבוע ב-$) או `percent` (אחוז מ-equity הנוכחי) |
| **סכום השקעה / אחוז** | התקציב הבסיסי לעסקה |
| **כניסה במחיר (¢)** | מחיר חוזה מקסימלי שמוכן לשלם (1¢-99¢ = 0.01$-0.99$) |
| **מינ׳ חוזים** | מינימום חוזים לעסקה — חייב להיות ≥ `order_min_size` של השוק |

#### בלוק "שוק"

| שדה | משמעות |
|------|---------|
| **אורך חלון** | `5m` או `15m` — איזה שוק לסחור בו (BTC Up/Down 5 דקות או 15 דקות) |
| **כיוון מועדף** | `Up` / `Down` / `signal` (לפי המלצת ה-TA) |

#### בלוק "DCA" (Dollar Cost Averaging — חלוקת כניסה)

| שדה | משמעות |
|------|---------|
| **הפעלת DCA** | אם פעיל — לא נכנסים בבת אחת |
| **מספר slices** | כמה כניסות חלקיות (למשל 4) |
| **מרווח בין סלייסים (שנ׳)** | זמן בין כניסה לכניסה |
| **DCA override** | מותר לעבור את מחיר ה-cap בכניסות מאוחרות אם החוזה זול יותר |

#### בלוק "Take Profit (TP)"

| שדה | משמעות |
|------|---------|
| **TP %** | יציאה כשהרווח על הפוזיציה עובר את הסף הזה (לפי ה-bid) |
| **Hold to resolution** | אם True — לא יוצאים מוקדם, מחזיקים עד סוף החלון |
| **Hold to resolution min price** | חוזה צריך להיות מעל ערך זה כדי "להחזיק" עד הסוף |
| **Stop Loss** | יציאה במחיר מתחת לסף מסוים |

#### בלוק "Loss Recovery" (שחזור הפסד)

| שדה | משמעות |
|------|---------|
| **הפעלה** | מכפיל סכום הכניסה אחרי הפסדים רצופים |
| **מכפיל בסיס** | למשל 1.5x אחרי כל הפסד |
| **תקרת מכפיל** | לא לעלות מעל X (למשל 4x) |
| **Step %** | האם להעלות step-wise או אגרסיבי |

#### בלוק "Hedge" (גידור)

| שדה | משמעות |
|------|---------|
| **הפעלה** | אם פעיל — קונה גם את הצד השני כש-Combined Ask < סף |
| **Combined Ask Max** | סף הגידור (לרוב סביב 0.96 — קונים את שני הצדדים כשהסכום שלהם נמוך מ-1.00$) |

#### בלוק "ביצוע הזמנה"

| שדה | משמעות |
|------|---------|
| **Order Mode** | `limit` (GTC) או `market` (FOK/FAK עם slippage cap) |
| **Entry Slippage %** | חריגת מחיר מקסימלית בכניסה |
| **Exit Slippage %** | חריגה מקסימלית ביציאה (FAK + retry ladder) |
| **Retry Max Attempts** | כמה ניסיונות חוזרים אם FAK חלקי |

#### פריסטים (Presets)

לחיצה על preset (כמו "מתחיל פשוט", "אגרסיבי") ממלאת את כל השדות בערכים ברירת-מחדל ידועים. הפריסטים מוגדרים ב-[App.tsx:`PRESETS`](../src/App.tsx).

#### שמירה

כפתור "שמור הגדרות" ([App.tsx:setSaveFeedback](../src/App.tsx)) שולח את כל
הקונפיג ל-`POST /api/strategy/config`, שמחזיר אישור ושומר ל-`config_persisted.json`
**ב-tmp+rename אטומי** (תיקון מהקומיט האחרון).

### 3.3 📡 סיגנלים (`signals`)

**מטרה:** ניתוח טכני (TA) מתקדם על מחיר BTC, ומה האלגוריתם ממליץ.

**קובץ:** [src/SignalsPanel.tsx](../src/SignalsPanel.tsx).

**מקורות:**
- `GET /api/signals` — מנוע ה-TA מחשב RSI, EMA, ATR, momentum, סנטימנט וכו'.
- מצרף ציון מסכם בין -1 ל-1, ועוצמת ביטחון באחוזים.

**מה רואים:**

1. **המלצה מסכמת** — `Up` / `Down` / `Neutral`, עם אחוז ביטחון.
2. **ציון משוקלל** — מספר בין -1 (חזק Down) ל-+1 (חזק Up).
3. **כרטיסי TA אינדיבידואליים:**
   - **RSI** (14, 1m): אם מתחת ל-30 → oversold (Up). מעל 70 → overbought (Down).
   - **EMA 9 vs EMA 21**: crossover.
   - **ATR (Average True Range)**: תנודתיות.
   - **Momentum 3m / 5m**: שינוי באחוזים בחלון.
   - **סנטימנט שוק** (clob_imbalance): יחס bid/ask מצטבר.
4. **טבלת היסטוריה** — סיגנלים אחרונים והאם צדקו (אם הסשן הסתיים).

**שימוש:** אם בחרת באסטרטגיה `side_preference=signal`, המנוע יסחור לפי
ההמלצה הזו (Up/Down) רק אם הביטחון מעל הסף שהגדרת.

### 3.4 ⚡ מסחר מהיר (`trigger`)

**מטרה:** מנוע טריגרים נפרד מה-StrategyRunner — כניסות מהירות לפי אירועים
חיצוניים מהירים יותר.

**קובץ:** [src/TriggerTrader.tsx](../src/TriggerTrader.tsx).
**מנוע backend:** [engine/trigger_engine.py](../engine/trigger_engine.py).

**שלושה מצבי טריגר** (נבחר אחד בכל פעם):

| מצב | תיאור |
|-----|--------|
| 🚀 **momentum** | BTC זז X% (למשל 0.2%) בתוך Y שניות (למשל 30s) → כניסה אוטומטית בכיוון התנועה (או בכיוון שהוגדר ידנית). |
| 📡 **signal** | ביטחון סיגנל TA עובר סף (למשל 70%) → כניסה לכיוון ההמלצה. |
| 🔄 **dca_pulse** | N סלייסים מהירים ברצף (למשל 5 כניסות במרווח של 10s). שימושי כשבטוחים בכיוון. |

**שדות משותפים:**
- **investment_usd**: סכום כל כניסה.
- **entry_price_cents**: cap מחיר.
- **take_profit_pct**: יציאה ברווח.
- **max_triggers_per_window**: לא להפעיל יותר מ-X פעמים בכל חלון של 5/15 דקות.
- **cooldown_sec**: זמן השהיה בין טריגרים רצופים.
- **min_seconds_remaining**: לא להיכנס כשנותרו פחות מ-X שניות לסיום החלון.
- **contract_max_drift_pct**: אל תיכנס אם החוזה כבר זז יותר מ-X% (מאוחר מדי).

**מצב Status** ([App.tsx:trigger.status_log](../src/App.tsx)):
לוג קצר של 50 הודעות אחרונות ("נסרק momentum...", "נכנס Up 5$@0.42", "cooldown 12s").

**הבדל מ-StrategyRunner:**
- StrategyRunner: ריצה רגילה לפי טווח, DCA איטי, TP, גידור — בנוי לסחר רציף.
- TriggerEngine: reactive, חד-פעמי לאירוע, מהיר. מתנתק אחרי שביצע (cooldown).

שניהם רצים במקביל; אפשר להפעיל אחד או את שניהם.

### 3.5 סטטיסטיקה (דמו) (`stats`)

**מטרה:** ביצועי הבוט במצב סימולציה.

**מה רואים** ([App.tsx:3880-4176](../src/App.tsx#L3880)):

1. **שורה עליונה (KPIs):** יתרה נוכחית, equity (יתרה + שווי פוזיציות פתוחות),
   PnL ממומש מצטבר, % win-rate, מספר עסקאות.

2. **גרף PnL לפי סשן** (PnlSessionAreaCharts):
   - **Open**: PnL לא ממומש לפוזיציות פתוחות לאורך זמן.
   - **Closed**: רווח/הפסד מצטבר של עסקאות סגורות.

3. **טבלת עסקאות** — כל עסקה (BUY/SELL) עם:
   - timestamp, צד, חוזים, מחיר, עמלה, PnL ממומש.
   - peak/trough של הפוזיציה (לזיהוי "החמצתי TP").
   - settlement BTC table: מה היה מחיר BTC בכניסה / יציאה / סוף חלון.

4. **קיבוץ לפי session** — אם פתחת פוזיציה ב-DCA של 4 סלייסים, מציג סשן אחד
   במקום 4 שורות נפרדות.

5. **פילטרים:** רק רווחים / רק הפסדים / הכל. expand-all.

### 3.6 סטטיסטיקה לייב (`stats_live`)

**הבדלים** ([App.tsx:3895-3927](../src/App.tsx#L3895)):

  כשהיומן הפנימי לא תאם לחלוטין את ה-CLOB.
- יתרה ושווי נטו מוצגים מ-`/api/live/portfolio` (CLOB ישיר), לא מ-`demo_state`.

### 3.7 ניתוח v3 (`tips_v2`)

**מטרה:** ניתוח מתקדם של ביצועי הבוט והמלצות אופטימיזציה.

**קובץ:** [src/TipsV2.tsx](../src/TipsV2.tsx).
**Backend:** [engine/tips_v2.py](../engine/tips_v2.py).

**שם הפנימי `tips_v2` אבל בתווית כתוב "ניתוח v3"** — מטעמי תאימות לאחור.

**מה זה עושה:**

1. **בוחר ריצות** (run folders ב-`logs/runs/<date>/<time>/`):
   - מתאים פילטרים: רק ריצות עם ≥ N עסקאות, רק שבוע אחרון וכו'.
2. **מחשב מטריקות:**
   - **Expectancy**: רווח ממוצע צפוי לעסקה.
   - **Win rate**, **Avg win**, **Avg loss**.
   - **RR Ratio** (Risk:Reward).
   - **Expire rate** (% עסקאות שהגיעו לפקיעת חלון בלי TP).
   - **Avg peak ROI** vs **TP target** — האם החמצנו רווחים כי TP היה נמוך מדי?
   - **Time buckets**: פירוק לפי דקת כניסה בחלון (אילו דקות הכי רווחיות).
3. **השוואת bins** — מבצע grid search על TP%, entry price, side preference,
   ומציע ערכים אופטימליים יחסית להיסטוריה שלך.
4. **bundles** — קבוצות תובנות מסוכמות שאפשר להעתיק.

## 5. זרימת מסחר טיפוסית

1. **gamma discovery**: כל 10s, market_discovery בודק שוק פעיל ב-Polymarket Gamma.
2. **WS subscription**: ws_price_stream נרשם ל-token_up + token_down של השוק.
3. **price updates**: WS משדר bid/ask → tp = TokenPrice → broadcast לכל לקוחות UI.
4. **strategy tick** (כל ~2s):
   - בדיקת מחיר חוזה מול cap.
   - בדיקת TA signal (אם side_preference=signal).
   - בדיקת זמן שנותר.
   - אם מתאים — pending entry נוצר.
5. **approval** (אם manual mode):
   - UI מציג pending עם פרטים.
   - משתמש לוחץ "אשר" → POST `/api/strategy/approve`.
6. **execution**:
   - **demo**: DemoEngine.simulate_market_buy → מעדכן state.
   - **live**: live_clob.place_entry_order → CLOB API; fill_price נשמר.
7. **monitoring**: כל tick בודק:
   - האם TP הושג?
   - האם stop loss?
   - האם DCA slice הבא?
   - האם הגיע סוף החלון?
8. **window settlement**: בסיום החלון, history_tracker.record_window_result
   מזהה מי ניצח (Chainlink) ומעדכן את ה-DB.
9. **reconcile** (כל 120s במצב לייב): סנכרון יומן פנימי מול יתרה אמיתית ב-CLOB.

## 7. נקודות לפיתוח עתידי (הצעות)

נשמר כאן כדי שנדע מה כבר נסקר:

- ⚠️ **CLOB min order size cache TTL** של 120s — שוק 5m מתחלף כל 300s; אחרי
  rollover יש חלון של עד דקתיים שבהם min_size cached עלול להיות מהשוק הישן.
- ⚠️ **WS unsubscribe** לטוקנים ישנים אחרי rollover — כרגע רק מוסיפים, לא מסירים.
- 🔧 **thread-safety** ל-caches גלובליים (klines, signals, persisted key) — אין lock.
- 🔧 **Frontend bundle** של 924KB — code splitting יקטין latency של טעינה ראשונית.
- 📊 **AnalyticsV3** עדיין משתמש בתת-לשוניות; יכול להיות חכם לאחד עם TipsV2 שאומר
  בעצם את אותו דבר במבנה אחר.
