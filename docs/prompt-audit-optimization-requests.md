## תפקידך

אתה עובד על הריפו **polymarket-bot-mac-v3**. המטרה בשלושה שלבים:

1. **ביקורת** — לזהות מה מתוכנית הביצועים/אופטימיזציה **עדיין לא בוצע** או חלקי (למשל: polling אגרסיבי, cache בשרת, רקע/טאב מוסתר, סערת בקשות ל־Railway).
2. **תכנון** — להציע וליישם שיפורים כך שהמערכת **תהיה מהירה**, אבל **לא תציף** את השרת או את ה־CLOB במצעד אחד (אין “צunami” של GET-ים; POST חשובים לא נתקעים מאחורי תור של GET).
3. **ביצוע** — שינויים בקוד/הגדרות בלבד לפי הצורך, עם בדיקות הגיוניות.

## כלים ו־“מספר Skills במקביל”

אל תסתמך על ניחוש. **פתח וקרא במקביל** (כמה הקשרים ביחד) לפחות:

- `.cursor/rules/railway-stream-performance.mdc` — דפוס הבעיה, קבצים מרכזיים, סף הצלחה.
- `engine/main.py` — cache TTL, נתיבי API כבדים.
- `src/api.ts` — timeout, `isPageHidden`.
- `src/App.tsx` — לולאות רענון / פולינג.
- `src/LiveStreamTrade.tsx` — שידור: מרווחים, orderbook, portfolio.

אם יש Skill רלוונטי בספריית המשתמש (למשל יצירת Skill / בדיקות ביצועים) — **השתמש בו** רק אם הוא מקצר עבודה; העיקר התוצאה בקוד.

## עקרונות לאופטימיזציה (גם אם המשתמש לא טכני)

- **אגרסיביות מול עומס:** הפחתת תדירות פולינג + **cache בצד שרת** לנתונים יקרים (למשל orderbook) עדיפה על “בקשה כל 200ms” מכל לקוח.
- **אצווה / payload גדול יותר:** אם אפשר — **נתיב API אחד** שמחזיר כמה שדות נדרשים ל־UI (snapshot) במקום שרשרת קריאות זהות; לא להמציא אנדפוינטים בלי צורך.
- **רקע:** כשהדף מוסתר — backoff או עצירת פולינג (כבר קיים חלקית — לוודא עקביות).
- **Railway:** משתני סביבה כמו `ORDERBOOK_CACHE_TTL` — לתעד אם משנים ברירות מחדל.

## קובץ משימות חובה — `docs/optimization-tasks.md`

- **לפני כל שינוי משמעותי:** עדכן את הקובץ `docs/optimization-tasks.md` בפרויקט.
- בכל עדכון: סמן משימות שבוצעו (`[x]`), השאר `[ ]` למה שנשאר.
- בסוף הקובץ יש בלוק **“סטטוס סופי”** — עדכן אותו רק כשהכל באמת הושלם.

כאשר **כל** המשימות בקובץ מסומנות כבוצעו — הוסף בשורה האחרונה של הקובץ (או ממש מתחת לבלוק הסטטוס) את השורה הבאה בדיוק:

```text
✅ הכל בוצע בהצלחה
```

אם משהו נחסם או דורש החלטת משתמש — רשום בקובץ תחת “חסמים” ואל תסמן הצלחה מלאה.
